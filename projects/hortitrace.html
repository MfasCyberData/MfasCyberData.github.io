Context and problem

HortiTrace runs a PostgreSQL data warehouse that feeds business reporting for about 6,425 logistics partners. The warehouse held 6.3 million transaction rows built up since 2010, and the ETL in place reloaded the whole dataset on every hourly cycle, roughly 30 minutes per run. Balance reports used every day by partners and the operations team scanned over 6 million rows on each run and took more than a minute to generate. One constraint shaped every decision: the work had to fit into a fixed, running stack without changing any source, report, or workflow, and it had to deploy without disrupting a service in daily use.

Approach

Both problems had the same root cause: loading and querying strategies that had not kept up with data growth. I addressed each layer on its own.

Incremental loading. I moved the ETL from a full-table reload to incremental loading on a 92-day rolling window, filtered on the transaction creation date. Incremental loading leaves the table in a partly updated state for a few minutes, so to keep reports from reading it mid-refresh I built a separate physical stable copy of the table, refreshed hourly by pgAgent with a staging-and-rename pattern. The volume reloaded per cycle dropped from 6.3M rows to about 312K (−95%), and cycle time from about 30 minutes to about 6 (−80%).

Pre-aggregated balance snapshot. For reporting, I built a snapshot layer inside the warehouse that pre-aggregates frozen historical balances up to a cutoff date. At each hourly refresh, pgAgent rebuilds the snapshot table by joining those pre-aggregated rows with the live recent transactions through UNION ALL, using the same staging-and-rename pattern. Reports read the resulting physical table directly. Report queries went from scanning 6.1M rows to about 530K (−91%), and generation time from over a minute to under 10 seconds (−83%).

Both new tables were registered in MyReport Data as new PostgreSQL data sources, alongside the original transaction model, which itself moved from full reload to incremental loading. The model used for reporting, original or one of the two new ones, is selected by pointing the warehouse at the right source, one at a time, with no change to any report. The external sources and every existing report stayed exactly as they were.

Constraints and trade-offs

Both solutions had to fit the tools already in place (MyReport as the ETL and reporting layer, the ERP as the source) and leave business processes untouched. Three decisions meant choosing between real alternatives.

Loading strategy. Full extraction was rejected because its cost grows with the total size of the table no matter what has changed, so it cannot keep up with data growth. The update/insert/delete mode was rejected because the text-typed primary key sorts differently in the ERP and in PostgreSQL, which makes deletions unreliable. Incremental loading on the creation date was chosen because the ERP's three-month immutability rule gives a provable boundary: no record outside the 92-day window can change, which permanently caps the reload at about 312K rows whatever the table grows to.

Reporting layer. Index tuning and query rewriting alone were rejected because they lower the cost of scanning a fixed volume, not the volume itself, so the gain fades as the table grows. Pre-aggregating frozen history into a physical snapshot removes the scan of the frozen 95% of the data entirely, a structural gain rather than an incremental one. A materialised view was the obvious tool for this, but a production constraint ruled it out: MyReport Data's weekly full rebuild drops and recreates the source table with CASCADE, which silently removes every dependent object, materialised views included. Only a fully independent physical table, with no dependency link to the source, survives that cycle.

Consistency mechanism. TRUNCATE then INSERT was rejected because it leaves the table empty for two to three minutes per refresh, visible to any report running at that moment. The staging-and-rename pattern keeps the inconsistency window to a single ALTER TABLE RENAME, about ten milliseconds, with an exception handler that restores the previous version if the swap fails, so the table can never be left in an undefined state.

Validation

All development and testing ran in a dedicated test environment, synchronised from production before each validation round. The solution only moved forward after passing a structured 7-test protocol: comparing partner-level and article-level balance totals across all 6,425 partners and about 37,500 partner-article combinations, and confirming that the sum of all business-filtered quantities across every partner came to zero. A Go / No-go decision was then taken with my supervisor. The SQL objects were deployed to production in dependency order, validated again against real production data, and the pgAgent jobs activated. The result on deployment day, 3 April 2026: zero balance differences, zero reports changed, no service interruption.

Results

Both solutions went live on 3 April 2026. Earlier in the internship, before the main project entered its supervised development phase in February 2026, I also held a day-to-day continuity role: daily ETL log monitoring, report and dashboard maintenance, and a set of development requests including a collection-threshold alert system, automated geographic distribution maps, and an invoicing-file automation.
